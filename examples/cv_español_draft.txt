Implementación de un proceso de parche automático para máquinas RHEL 5-7 con auditoría / atraso
Proveedor de servicios financieros
- Al estar sujeto a los estándares de seguridad como PCI-DSS, un proveedor alemán de servicios financieros contactó a mi compañía para ayudar a automatizar sus procedimientos de parches.
- Varias herramientas tuvieron que ser integradas / orquestadas: RedHat Satellite 5.x (Herramienta de administración de parches), Atlassian Jira (sistema de tickets / auditoría), Base de datos nacional de vulnerabilidades (nvd.nist.gov).
- Creación de un puente a nvd.nist.gov para clasificar mejor los parches de seguridad de RedHat contra el puntaje CVSS básico del NVD debido a los requisitos de PCI-DSS
- Desarrollo de un puente desde Satellite(¿?) en el sistema de emisión de tickets. El puente automáticamente abre nuevos tickets con sub-tareas especificadas (según demandas del cliente). El puente realiza un seguimiento automático del proceso de parchado, actualiza los tickets a medida que los sistemas se aplican y cierra automáticamente cada ticket específico tan pronto como se completa el parcheo (¿?) de una vulnerabilidad específica.
- Además, se tuvo que crear easy to use text-mode frontend para que el ingeniero de sistemas pueda iniciar el patching (was ist das??) por sí mismo.
- Tras la implementación de este proyecto, el tiempo total necesario para gestionar el proceso de patching mensual se redujo de aproximadamente 2-3 días a unos 5 minutos.
- Tecnologías usadas: RedHat Satellite 5.x and its APIs, Python 2.7, Atlassian Jira APIs, nvd.nist.gov XML-dumps, bash

Implementación de un sistema de implementación automática para Windows 7 y recursos de capacitación técnica basados en Linux / Bittorrent
Proveedor de capacitaciones técnicas
- El proveedor de capacitación en el que trabajé necesitaba un mejor proceso para implementar los materiales de capacitación / máquinas virtuales y la configuración necesaria para los nuevos cursos. Como hay diferentes cursos con sus propios requisitos cada semana, cada estación de trabajo debe implementarse al menos una vez a la semana.
- Para evitar las limitaciones de Windows y su proceso de implementación (despliegue?), se utilizó una nueva característica (en ese momento) de Windows: Arranque directo de WMI-Files
- Con la ayuda de un sistema personalizado de Linux en vivo iniciado a través de PXE, se desplegará una nueva imagen de Windows en la máquina.
- El sistema en vivo obtiene una lista de los recursos necesarios para el próximo curso desde una aplicación central (desarrollada en PHP)
- A continuación, el material de curso relevante (con la descarga de peer2peer vía bittorrent) se copia a la máquina local. El uso de bittorrent es crítico aquí porque varios cursos técnicos de Microsoft son enormes en términos de tamaño de archivo (VM Images, Setups, ISO Files)
- Después del primer inicio, varios scripts basados en VBScript y Powershell configuran la máquina para support (die veste Übersetzung suchen) el curso de las próximas semanas. Las políticas de grupo manejan la configuración genérica de las máquinas.
- Tecnologías usadas: Debian preseed/liveiso tools, bash, aria2c/bittorrent, Windows WMI Image format, VBS, Powershell, Active Directory Group Policies, PHP
